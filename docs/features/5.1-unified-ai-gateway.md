# 5.1. Unified AI Gateway (Lớp Gateway AI Hợp Nhất)


**Status:** Partial Baseline / Planned Hardening
**Owner docs:** `docs/features/5.1-unified-ai-gateway.md` (this file); `docs/ARCHITECTURE.md`
**Code areas:** `server/internal/gateway`, `server/pkg/llm`, `server/migration/000011_unified_ai_gateway.*.sql`, `web/src/` provider settings UI
**Blocking decisions:** Final REST API shape for credentials, frontend provider settings UX, audit-log coverage.
**Acceptance criteria:** Admin can configure multiple provider credentials with model list per Model Level Group (Fast/Balanced/Powerful), route requests by level, and observe usage/cooldown events.

**Mục tiêu:** Cung cấp một lớp gateway tập trung quản lý **định tuyến model, budget tracking, format translation và fallback tự động** — thay thế cách tiếp cận đơn giản "lưu API key + chọn model trực tiếp" trước đây. Agent và client chỉ gọi vào gateway với một route ổn định, không cần biết provider thật phía sau.

> **Tiến độ triển khai:**
> - ✅ **Hoàn thành Backend Core (Baseline):** Đã implement Budget tracking (atomic), Multi-key pool rotation (fill-first/round-robin), Cooldown recovery.
> - ✅ **Tích hợp Orchestrator (Baseline):** Gateway đã được gắn vào pipeline (fallback tự động sang biến môi trường nếu DB trống).
> - ✅ **Baseline REST API & UI:** Đã triển khai cơ bản việc quản lý provider credentials routes/UI.
> - ⏳ **Kế hoạch (Planned Target):** Format Translation, Token Saver, và UX danh sách model nâng cao. (Lưu ý: Virtual Key routing đã được hoãn lại ở phần F).


> **Nguyên lý cốt lõi:** Agent không bao giờ gọi thẳng OpenAI/Anthropic/Gemini. Agent gọi vào internal gateway có contract cố định. Gateway chịu trách nhiệm resolve model, dịch format, chọn credential, fallback và tracking.

---

#### A. Kiến trúc Hiện tại (Current Baseline)

**1. Quản lý Provider Credentials (Multi-Key)**
Hệ thống hỗ trợ lưu trữ **nhiều API key** cho cùng một Provider.
*   **Lưu trữ an toàn:** API Key được mã hoá **AES-256-GCM** và lưu trong bảng `provider_credentials`. (Lưu ý: Legacy `.env` chỉ là fallback khi DB chưa được cấu hình).
*   **Credential Selection Strategy:** Tại runtime, Gateway tự chọn credential phù hợp theo chiến lược `fill-first` (dùng key đầu tiên cho đến khi hết quota) hoặc `round-robin` (luân phiên giữa các key).
*   **Cooldown & Recovery:** Key bị rate limit/quota sẽ tự động bị loại khỏi vòng lặp (auto-exclude) và đưa vào trạng thái cooldown. Khi hết thời gian cooldown, hệ thống tự động unlock. Lợi ích: Không bao giờ gián đoạn task khi 1 key hết quota.

**2. Phân nhóm Model (Model Level Groups)**
Thay vì Agent bị trói buộc vào một model cụ thể (VD: `gpt-4o`), hệ thống cấu hình các danh sách Model theo các **Model Level Group**:
*   **Fast:** Dành cho các task đơn giản, cần tốc độ trả lời cực nhanh (VD: `gpt-4o-mini`, `gemini-2.5-flash`).
*   **Balanced:** Dành cho hầu hết các task lập trình, cân bằng tốt giữa tốc độ và khả năng lý luận.
*   **Powerful:** Dành cho các task phức tạp, đòi hỏi khả năng tư duy sâu sắc, kiến trúc hệ thống, hoặc review (VD: `claude-opus`, `gpt-4o`).

*Lưu ý:* Khi thêm một Provider, hệ thống sẽ tự động tạo một danh sách các model mặc định chia vào 3 nhóm này. **Admin có toàn quyền thêm mới, chỉnh sửa hoặc xóa bỏ model khỏi từng nhóm**.

**3. Model Routing Cơ Bản**
*   Khi cấu hình, Agent chỉ cần thiết lập Level (VD: "Balanced"). 
*   Gateway sẽ tra cứu danh sách model được phép sử dụng trong nhóm Balanced, kết hợp với các Provider Key đang có sẵn (và không bị rate-limit) để thực hiện cuộc gọi.

#### B. Kiến trúc Kế hoạch (Planned Target)

**1. Dynamic Fallback Chain (within Model Level Group)**
*   Thay vì cấu hình các combo tĩnh phức tạp, Gateway tự động xây dựng chuỗi dự phòng động dựa trên danh sách model trong cùng một Level Group (`fast`, `balanced`, `powerful`) của tổ chức, sắp xếp theo thứ tự `priority` (ví dụ: `Priority 1: gemini-1.5-pro -> Priority 2: gpt-4o-mini -> Priority 3: claude-3-5-haiku`).
*   Gateway tự động fallback theo chuỗi này nếu model có độ ưu tiên cao hơn bị sập hoặc hết quota, đảm bảo zero downtime.


**2. Format Translation & Token Saver**
*   **Format Translation:** Tự động dịch định dạng input/output giữa các nhà cung cấp khác nhau (OpenAI Chat ↔ Claude Messages ↔ Gemini).
*   **Token Saver:** Tự động phát hiện các đoạn text dài không cần thiết (VD: output của git diff, build logs) và nén nội dung lại để tiết kiệm 20-40% input token trước khi gửi đi.

#### C. Chi tiết Luồng Vận Hành

**1. Luồng Quản trị (Admin Flow)**
```
Bước 1: Quản lý Provider & API Keys
        ├── Thêm Provider mới.
        ├── Nhập 1 hoặc NHIỀU API Key cho provider đó.
        └── Chọn chiến lược xoay vòng (fill-first | round-robin) và base URL tuỳ chỉnh nếu có.

Bước 2: Quản lý danh sách Model (Model Level Group)
        ├── Tự động/thủ công tạo danh sách model thuộc các level: Fast / Balanced / Powerful.
        └── Thêm mới, chỉnh sửa (edit), hoặc gỡ bỏ model khỏi các group tùy theo nhu cầu tổ chức.
```

**2. Luồng Thực thi (Agent Runtime Flow)**
```
Bước 1: Agent khởi tạo Request
        └── Agent được cấu hình với một Model Level nhất định (Ví dụ: "Balanced").

Bước 2: Gateway Phân giải (Resolve)
        ├── Lấy danh sách các available models thuộc Level "Balanced" kèm theo các API key tương ứng.
        └── Resolve model ưu tiên nhất.

Bước 3: Gateway Xử lý & Gọi LLM
        ├── Chọn credential tốt nhất từ pool (bỏ qua các key đang bị rate-limited).
        ├── (Planned) Format Translation & Token Saver.
        ├── Execute gọi API của Provider.
        └── Xử lý lỗi linh hoạt (nếu lỗi rate-limit: đưa key vào cooldown và thử lại key/provider khác).

Bước 4: Ghi nhận (Telemetry & Audit)
        └── Lưu usage logs (provider, model, key_label, tokens, cost, latency).
```

---

#### D. Bảo mật & Lưu trữ
*   Real API Key mã hoá **AES-256-GCM** — không bao giờ trả plaintext về frontend hay ghi log.
*   Mỗi Organization sở hữu một không gian key + model config hoàn toàn cách ly.
*   Backend validate chặt chẽ: reject provider không nằm trong allowlist, từ chối model route không hợp lệ.

#### F. Kế hoạch xa (Distant Future / Hoãn lại)

Tính năng **Virtual Key Architecture** hiện đang được hoãn lại và không nằm trong kế hoạch phát triển gần.
*   **Khái niệm:** Cấp phát "Virtual Key" (VD: `sk-aco-...`) cho từng Agent, Project hoặc User để sử dụng thay cho API key thật. Virtual Key sẽ được hash with salt để verify và audit log toàn bộ lifecycle.
*   **Lợi ích:** Virtual Key cho phép quản lý **budget cap** (hạn mức chi tiêu) và **rate limit** riêng biệt, tránh việc một Agent bị loop vô hạn làm cạn kiệt tài nguyên của tổ chức. Khi được phát triển, Virtual Keys sẽ lưu trong bảng `virtual_keys`. Lúc đó Request từ Agent sẽ đi kèm với việc Gateway trừ Budget trước khi Execute LLM.
