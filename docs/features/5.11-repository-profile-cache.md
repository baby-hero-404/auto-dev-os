# 5.11. Repository Profile Cache (Bộ Nhớ Ngữ Cảnh Repository)

**Status:** Partially Implemented (cache load/generate in context_load, invalidation planned)  
**Owner docs:** `docs/ARCHITECTURE.md`, `docs/features/5.7-workflow-engine.md`  
**Code areas:** `server/internal/orchestrator/steps/context_load.go` (profile loading, generation, caching), `server/.data/repositories/<repo_hash>/profile.json` (cache storage)  
**Acceptance criteria:**  
1. Khi bước `Context Load` phát hiện thiếu tài liệu (`ARCHITECTURE.md`, `CONTRIBUTING.md`, quy chuẩn linter), hệ thống tự động khởi tạo Agent phân tích cấu trúc mã nguồn.
2. Kết quả phân tích (cấu trúc thư mục, domain logic, convention mặc định) được lưu và cache bên ngoài tại thư mục của server (`server/.data/repositories/<repo_hash>/profile.json`).
3. Khóa cache (cache key) được tạo từ thông tin repository (ví dụ: SHA-256 hash của normalized URL) để có thể chia sẻ tri thức giữa nhiều Project sử dụng chung repository.
4. Trong các lần thực thi tiếp theo, bộ nhớ cache này được nạp trực tiếp vào ngữ cảnh LLM mà không cần gọi lại Agent phân tích hay thay đổi code trong workspace.

---

## A. Tổng Quan & Mục Tiêu

Để Agent viết code và phân tích yêu cầu chính xác, bước `Context Load` (§5.7 Step 0) cần nạp đầy đủ thông tin về cấu trúc thư mục, quy tắc thiết kế và coding conventions của mã nguồn hiện tại. 

Tuy nhiên, phần lớn các repository mới hoặc chưa hoàn thiện tài liệu thường thiếu các tệp như `ARCHITECTURE.md` hay `.golangci.yml`. Để giải quyết vấn đề này mà không làm ảnh hưởng (gây ô nhiễm Git commit) đến repository gốc, hệ thống triển khai cơ chế **Repository Profile Cache**:

```text
                  Context Load (Step 0)
                            │
               ┌────────────┴────────────┐
       [Tìm thấy tài liệu?]      [Thiếu tài liệu?]
               │                         │
               ▼                         ▼
        Nạp trực tiếp           Kiểm tra Server Cache
                                         │
                         ┌───────────────┴───────────────┐
                  [Có trong Cache?]             [Chưa có trong Cache?]
                         │                               │
                         ▼                               ▼
                   Nạp từ Cache                  Khởi chạy Profiling Agent
                                                 (Phân tích cấu trúc codebase)
                                                         │
                                                         ▼
                                                Lưu vào Server Cache
                                           (repositories/<repo_hash>/profile.json)
```

## B. Thiết Kế Chi Tiết

### 1. Cơ Chế Khóa Cache (Cache Key)
Để chia sẻ tri thức một cách thông minh, cache được tổ chức theo **Repository URL** thay vì `project_id`.
*   **Normalized URL**: Chuẩn hóa URL của repository bằng cách loại bỏ giao thức (`http://`, `https://`), user-info (token, mật khẩu), và đuôi `.git`.
    *   Ví dụ: `https://github.com/user/my-repo.git` -> `github.com/user/my-repo`
*   **Repo Hash**: Sử dụng SHA-256 hash của normalized URL làm tên thư mục lưu trữ cache trên server.
    *   Thư mục lưu trữ: `server/.data/repositories/<repo_hash>/profile.json`

### 2. Cấu Trúc Dữ Liệu `profile.json`
Tệp cache chứa thông tin đúc kết từ repository bao gồm:
```json
{
  "repo_url": "github.com/user/my-repo",
  "generated_at": "2026-06-24T15:30:00Z",
  "architecture": {
    "summary": "Tổng quan về kiến trúc của repository (ví dụ: Go clean architecture, NextJS app router).",
    "directory_structure": {
      "server/": "Chứa mã nguồn Go backend.",
      "web/": "Chứa mã nguồn Next.js frontend."
    }
  },
  "conventions": {
    "language": "Go & TypeScript",
    "naming": "CamelCase cho Go structs, camelCase cho biến JS.",
    "linter_rules": "Không sử dụng any trong TypeScript, xử lý tất cả lỗi trả về trong Go."
  },
  "key_modules": [
    {
      "name": "Orchestrator",
      "path": "server/internal/orchestrator",
      "description": "Điều phối vòng đời công việc và giao tiếp AI Agent."
    }
  ]
}
```

### 3. Quy Trình profiling và Sinh Cache
Khi cache bị thiếu (Cache Miss):
1. **Quét cấu trúc thư mục (File Tree Scan)**: Đọc cấu trúc cây thư mục của repository gốc (giới hạn độ sâu tối đa là 3 cấp).
2. **Đọc tệp cấu hình**: Tự động phân tích các tệp đặc trưng (`go.mod`, `package.json`, `Cargo.toml`, `requirements.txt`) để xác định ngôn ngữ lập trình chính và các thư viện quan trọng.
3. **Phân tích thông minh (LLM Profiling)**: Gửi thông tin cây thư mục và cấu hình trên tới một Agent phân tích nhanh (Lightweight Profiling Agent). Agent này đúc kết ra tài liệu kiến trúc cơ bản và lưu lại vào `profile.json`.

## C. Điểm Cộng & Điểm Trừ (Trade-offs)

*   ✅ **Ưu điểm**:
    *   **Giữ sạch Git tree**: Không tự động sinh tệp lạ và commit vào nhánh làm việc của người dùng.
    *   **Chia sẻ bộ nhớ (Cross-Project Recall)**: Nếu nhiều dự án liên kết tới cùng một Repository, chúng sẽ dùng chung một cache, tiết kiệm tối đa chi phí gọi LLM phân tích ban đầu.
    *   **Hiệu năng cao**: Sau lần nạp đầu tiên, các task tiếp theo sẽ nạp thông tin kiến trúc ngay lập tức từ file hệ thống cục bộ trên server.
*   ❌ **Nhược điểm**:
    *   **Tránh lệch cấu trúc (Stale Cache)**: Nếu mã nguồn thay đổi cấu trúc thư mục lớn, cache có thể bị lỗi thời. Cần có cơ chế phát hiện (ví dụ: kiểm tra commit hash mới nhất của default branch) để xóa và dựng lại cache.

## D. Kế Hoạch Triển Khai (Execution Plan)

1. **Giai đoạn 1: Thiết kế Cơ chế Hash & Lưu Trữ**
   * Triển khai hàm chuẩn hóa URL và SHA-256 hash tại `server/internal/gitops`.
   * Định nghĩa cấu trúc `profile.json` dưới dạng struct Go.

2. **Giai đoạn 2: Phát triển Profiling Agent**
   * Xây dựng prompt phân tích cấu trúc thư mục và sinh profile.
   * Viết logic tích hợp vào `server/internal/orchestrator/step_context_load.go` để chạy Agent này khi phát hiện thiếu tệp tài liệu và không có cache.

3. **Giai đoạn 3: Cơ chế Hết Hạn Cache (Invalidation)**
   * Lưu commit hash mới nhất tại thời điểm sinh cache.
   * So sánh commit hash hiện tại của repository với commit hash trong cache, nếu khác nhau thì thực hiện làm mới (refresh) cache.
