# 5.12. Patch Engine Abstraction

**Status:** Implemented  
**Owner docs:** `docs/ARCHITECTURE.md`  
**Code areas:** `server/internal/orchestrator/patch/` (engine, applier, validator, search_replace, helpers, types)  
**Acceptance criteria:**
- Patch application is decoupled from code generation via the `PatchApplier` interface.
- Multiple strategies are supported: Unified Diff (`git apply`) and Search & Replace (Aider-style block parsing).
- All patches pass through `PatchValidator` (syntax check, metadata validation, block uniqueness) before execution.
- In-step self-healing retry (up to 2-3 attempts) for format/parse errors before escalating to the workflow DAG.

**Mục tiêu:** Tách rời (decouple) quá trình tạo mã (code generation) của LLM khỏi quá trình áp dụng mã đó lên hệ thống file. Thay vì hardcode phụ thuộc hoàn toàn vào lệnh `git apply` truyền thống, lớp trừu tượng này giới thiệu một bước Validation (Xác thực) và hỗ trợ linh hoạt nhiều chiến lược vá lỗi khác nhau nhằm tối đa hóa độ ổn định và hỗ trợ tốt nhất cho vòng lặp tự phục hồi (self-healing).

---

## A. Patch Validator (Bộ Xác Thực)

Trước khi bất kỳ bản patch nào được thực thi lên file system, nó phải vượt qua **Patch Validator**. Thành phần này đóng vai trò như một bức tường lửa (firewall) nghiêm ngặt.

**Các bước kiểm tra (Validation Checks):**
- **Repository Existence:** Đảm bảo repository đích và worktree tồn tại.
- **Syntax Check:** Xác minh định dạng của patch (dù là Unified Diff hay Search & Replace) có đúng chuẩn cú pháp không.
- **Metadata Validation:** Đối với Unified Diff, đảm bảo các hunk headers (`@@`) khớp hoàn toàn về mặt toán học với số lượng dòng hiện tại của file đích.
- **Uniqueness Check:** Đối với khối Search & Replace, đảm bảo khối `SEARCH` chỉ khớp chính xác *một* vị trí duy nhất trong file đích để tránh ghi đè nhầm.
- **File Existence:** Ngăn chặn việc sửa đổi các file không tồn tại trừ khi có lệnh tạo file (create) rõ ràng.

## B. Patch Engine Interface (Giao Diện Chuẩn Hóa)

Một interface thống nhất bằng Go (`PatchApplier`) giúp chuẩn hóa cách áp dụng patch, bất kể chiến lược bên dưới là gì.

```go
type PatchApplier interface {
    Apply(ctx context.Context, task *models.Task, patchData string, targetPath string) error
    Validate(patchData string) []ValidationError
}
```

## C. Pluggable Strategies (Các Chiến Lược Có Thể Lắp Ghép)

Engine hỗ trợ linh hoạt nhiều chiến lược dựa trên định dạng output được parse từ LLM:

1. **Unified Diff (Legacy/Standard):**
   - Thực thi qua lệnh `git apply` truyền thống.
   - Độ tương thích với Git cao nhưng rất nhạy cảm với số dòng (line numbers).
2. **Search & Replace (Mô hình Aider — Robust Strategy):**
   - Sử dụng các khối `<<<<<<< SEARCH ... ======= ... >>>>>>> REPLACE`.
   - Hoạt động cực kỳ bền bỉ (robust) trước các lỗi trôi số dòng (line number drift) và rất tự nhiên đối với cách LLM sinh văn bản.
   - Tách làm 3 lớp rõ rệt:
     - **Parser:** State machine đọc luồng văn bản từ LLM, trích xuất cấu trúc và tạo ra danh sách `EditBlock` (có hỗ trợ mở rộng metadata như File: hay Action:).
     - **Validator:** Kiểm tra tính hợp lệ độc lập (file đích tồn tại, đoạn SEARCH xuất hiện chính xác 1 lần duy nhất trong file, không chồng chéo các khối sửa đổi).
     - **Applier:** Thực hiện strings.Replace và ghi dữ liệu an toàn xuống đĩa.
3. **AST Editing (Định hướng tương lai — Đã đưa vào Backlog):**
   - Xem chi tiết tại `docs/backlog/ast-editing-backlog.md`.

## D. Luồng Dữ Liệu & Khả Năng Tự Phục Hồi (Workflow Data Flow & Self-Healing)

Quy trình xử lý patch và lỗi được chia làm hai cấp độ rõ rệt:

### 1. In-Step Retry (Cho các lỗi định dạng/LLM Output)
Áp dụng trực tiếp trong các Coding Steps (`code_backend`, `code_frontend`, `fix`) để sửa lỗi format mà không cần chạy lại toàn bộ workflow hoặc cần Planner can thiệp.

```text
       [LLM Agent Sinh Patch]
                 │
                 ▼
       [Lớp Xác Thực (Validate)]
                 │
        ┌────────┴────────┐
     [PASS]            [FAIL] (Lỗi format, trôi dòng, trùng khớp, v.v.)
        │                 │
        ▼                 ▼
   [Apply Patch]     [Phản hồi lỗi lại cho LLM (Self-Healing)]
                          │
                          ▼
                 [LLM Sinh Lại Patch] (Tối đa 2-3 lần)
                          │
                          ▼
                 [Vẫn thất bại sau N lần]
                          │
                          ▼
                 [Trả về StepResult với Warning & PatchApplyError]
```

*Các lỗi áp dụng cơ chế In-Step Retry:*
- Lỗi cú pháp Unified Diff hoặc Search/Replace.
- Thiếu tiền tố repository (`Missing repo prefix`).
- Đoạn `SEARCH` khớp nhiều vị trí hoặc không khớp vị trí nào (`Search block matches multiple locations/not found`).
- Các lỗi format JSON, syntax do LLM format output.

### 2. Workflow-Level Retry (Cho các lỗi logic/nghiệp vụ)
Các lỗi logic phức tạp yêu cầu phải chuyển qua các step tiếp theo trong DAG (như Reviewer, FixStep, Planner) để điều chỉnh kế hoạch hoặc thiết kế lại:
- Test suite chạy thất bại (Tests FAIL).
- LLM Reviewer từ chối (Review FAIL).
- Nhầm lẫn business logic hoặc thiết kế sai.

## E. Ưu Điểm (Advantages)
- **Tách biệt Trách nhiệm (Separation of Concerns):** Tách rõ ràng Parser, Validator và Applier giúp mã nguồn dễ bảo trì, dễ viết unit test độc lập.
- **Tối ưu hóa Latency:** Giảm chu kỳ chạy của Workflow DAG nhờ tự phục hồi lỗi format nhẹ ngay trong Step.
- **Tính linh hoạt & Khả năng mở rộng (Flexibility & Extensibility):** Dễ dàng cắm thêm các Applier mới (như AST Edit trong tương lai) mà không thay đổi logic điều phối của workflow.
