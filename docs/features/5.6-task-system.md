# 5.6. Task System

**Status:** ✅ Baseline Implemented / ⏳ Planned Integrations
**Owner docs:** `docs/ARCHITECTURE.md`  
**Code areas:** `server/internal/service/task.go`, `server/internal/repository`, `server/pkg/models`, `web/src/` task screens  
**Blocking decisions:** External issue tracker priority: Jira, Linear, GitHub Issues, Notion.  

**Acceptance criteria:**
- [x] Developer can create tasks with repository context.
- [x] Task lifecycle states are persisted.
- [x] Orchestrator can advance task status through context load/analyzing/coding/review/testing/pr_ready/human review.
- [x] Task only completes after explicit human approve + merge (PR ≠ task done).

**Mục tiêu:** Cung cấp cơ chế để nhà phát triển mô tả yêu cầu (task) và giao cho AI thực thi tự động. Task là đơn vị công việc cốt lõi — mọi thứ trong hệ thống đều xoay quanh task.

## Cách Task Hoạt Động (6 Bước)

```
Bước 1: Tạo Task
   Con người tạo task mới với mô tả chi tiết yêu cầu.

Bước 2: Nạp Ngữ Cảnh (Context Load)
   Agent checkout repo, đọc cấu trúc dự án, conventions, CI config, test commands,
   ARCHITECTURE.md — đảm bảo đủ ngữ cảnh trước khi phân tích.

Bước 3: AI Phân Tích + Spec + Complexity
   Agent Planner phân tích task → tạo spec (scope, affected files, risks, test strategy)
   → đánh giá complexity (Easy/Medium/Hard) → xác định risk domains.

Bước 4: Human Gate (có điều kiện)
   • Easy + Low-risk → auto-approve spec.
   • Easy + High-risk (auth, payment, data migration, security...) → DỪNG chờ con người.
   • Medium/Hard → LUÔN dừng chờ con người review spec, plan, risks.

Bước 5: AI Thực Thi → Test → PR
   Plan execution → Code (có thể song song theo ownership) → Agent review chéo
   → Fix (bounded: max N vòng) → Full test + lint + build → Tạo PR.
   Task chuyển sang pr_ready — CHƯA hoàn thành.

Bước 6: Con Người Duyệt & Merge
   Con người review PR cuối cùng → Approve → Merge → Task hoàn thành (merged).
   Reject → Task quay lại fixing hoặc chuyển failed.
```

## Vòng Đời Task (12 Trạng Thái)

```
todo → context_loading → analyzing → spec_review → coding ⟷ reviewing ⟷ fixing → testing → pr_ready → human_review → merged
                                                                                                                      ↘ failed
```

| Phase | Status | Mô tả |
|:------|:-------|:------|
| 📋 **Planning** | `todo` | Task vừa được tạo, chưa xử lý |
| | `context_loading` | Agent đang checkout repo và nạp ngữ cảnh dự án |
| | `analyzing` | Agent Planner đang phân tích, tạo spec, đánh giá complexity + risk |
| | `spec_review` | Con người review spec/plan. Easy + low-risk → auto-skip; Easy + high-risk hoặc Medium/Hard → chờ approve |
| ⚡ **Executing** | `coding` | Agent đang viết code (có thể song song theo ownership) |
| | `reviewing` | Agent Reviewer đang review chéo code |
| | `fixing` | Agent đang sửa lỗi từ review (bounded: max `max_review_fix_cycles` vòng) |
| | `testing` | Agent đang chạy full test suite + lint + build |
| ✅ **Approval** | `pr_ready` | PR đã được tạo, chờ human review. Task **chưa** hoàn thành |
| | `human_review` | Con người đang review PR |
| | `merged` | Hoàn tất — PR đã merge, task hoàn thành |
| ❌ **Terminal** | `failed` | Lỗi xảy ra ở bất kỳ bước nào |

> PR ≠ Task hoàn thành. Task chỉ chuyển `merged` khi con người explicitly approve PR và merge thành công.
> Task ở trạng thái `failed` có thể được con người reset về `todo` để chạy lại.

## Thiết kế OpenSpec (Proposed Task Specification)

Khi Task chuyển sang trạng thái `analyzing`, hệ thống sẽ giao cho **Agent Planner** tiến hành khảo sát mã nguồn và tạo ra bản Đặc tả kỹ thuật đề xuất (**Proposed Task Specification**) theo chuẩn **OpenSpec**. 

### 1. Mục đích của OpenSpec trong quy trình tự động hóa
* **Human-in-the-Loop (Cổng kiểm soát của con người)**: Trước khi AI tiến hành code thực tế (tốn nhiều token và có nguy cơ lỗi/bảo mật), con người cần có một bản thiết kế chi tiết để xem trước. Bạn có thể sửa trực tiếp thiết kế này hoặc phê duyệt nó.
* **Hợp đồng thực thi cho Code Agents**: Bản đặc tả này hoạt động như một hợp đồng. Các Agent viết code (Backend/Frontend) ở giai đoạn `coding` sẽ chỉ đọc danh sách yêu cầu (`specs.md`) và danh sách công việc (`tasks.md`) để triển khai chính xác những gì đã được phê duyệt.
* **Tính độc lập và bền vững**: Các file đặc tả được ghi trực tiếp vào đĩa trong thư mục thay đổi của task: `openspec/changes/<change_name>/`. Nếu hệ thống bị tắt hoặc khởi động lại, nó có thể đọc lại đặc tả này để tiếp tục chạy (Resume).

### 2. Cấu trúc chi tiết của một Bản đặc tả OpenSpec
Một thư mục đặc tả hoàn chỉnh bao gồm 5 file chính:

| File | Tên File | Mục đích & Mô tả |
| :--- | :--- | :--- |
| 1 | `proposal.md` | **Đề xuất cấp cao**: Tổng quan về mục tiêu thay đổi, bối cảnh bài toán và giải pháp khái quát. |
| 2 | `specs.md` | **Đặc tả yêu cầu chi tiết**: Liệt kê các chức năng được thêm mới (`ADDED`), chỉnh sửa (`MODIFIED`), hoặc loại bỏ (`REMOVED`). Giúp lập trình viên và AI hiểu rõ hệ thống sẽ có thêm những hành vi gì. |
| 3 | `design.md` | **Thiết kế kỹ thuật**: Kiến trúc phần mềm, các Goals/Non-Goals, quyết định lựa chọn thư viện/kỹ thuật (Decisions), đánh giá rủi ro (Risks & Trade-offs) và câu hỏi mở cần làm rõ. |
| 4 | `tasks.md` | **Checklist đầu việc**: Phân rã dự án thành các việc nhỏ cần làm (có checkbox `- [ ]`). Các Agent sẽ dựa vào đây để code và tích dấu hoàn thành sau mỗi bước. |
| 5 | `.openspec.yaml` | **Metadata**: File cấu hình chứa định danh task (`taskId`), tên thay đổi (`changeName`), và trạng thái đặc tả (`status` - ví dụ: `pending_review`, `approved`). |

## Cơ Chế Multi-Repo Workspace

Hệ thống Task hỗ trợ linh hoạt việc thực thi trên một hay nhiều Codebase (Multi-Repo):

- **Single-Repo Task (Mặc định):** Khi một Task được gắn với một `RepositoryID` cụ thể, AI Sandbox chỉ chứa mã nguồn của Repository đó. Quá trình kiểm tra lỗi (`git diff`), tạo branch và Pull Request chỉ diễn ra cục bộ trong scope của Repo mục tiêu.
- **Multi-Repo / Cross-Repo Task:** Khi một Task không gắn `RepositoryID`, nó có scope là toàn bộ Project. Orchestrator sẽ tự động clone tất cả Repositories của Project vào các thư mục con bên trong Sandbox. Khi Task hoàn thành, Orchestrator quét thay đổi trên từng thư mục con, tiến hành tạo Pull Request độc lập cho **mỗi Repository có chứa thay đổi**. Một Task có thể sinh ra nhiều `pr_urls` và theo dõi chúng đồng thời.

## Cấu Trúc Workspace & Khả Năng Resume

Hệ thống quản lý vòng đời thực thi của từng task thông qua một cấu trúc workspace bền vững, chuyên biệt trên filesystem. Mỗi task sở hữu một workspace độc lập chứa toàn bộ source checkouts, generated specs, checkpoints, logs, và metadata:

### 1. Kiến trúc Thư mục Tiêu chuẩn

```text
workspaces/tasks/{task_id}/
├── .workspace.lock          # Khóa độc quyền (chống concurrent worker)
├── task.json                # Snapshot của database state (title, status, complexity)
├── metadata.json            # Cấu hình repo, paths, branch routing và integration status
├── specs/                   # Original prompt và phân tích ban đầu
├── openspec/
│   └── changes/
│       └── {change_name}/   # Tài liệu OpenSpec (proposal, specs, design, tasks, .openspec.yaml)
├── context/                 # Repo summary, conventions, architecture.md
├── code/
│   └── repos/               # Mã nguồn (hỗ trợ nhiều repo trong cùng 1 task)
│       └── {repo_name}/
│           ├── main/        # Integration checkout (nhánh feature/{task_id})
│           └── worktrees/
│               ├── {role}/  # Worktree cho từng agent role (backend, frontend...)
│               └── fix/     # Worktree để agent sửa lỗi sau review
├── artifacts/
│   ├── checkpoints/         # File JSON (kèm commit_hash) đánh dấu trạng thái sau mỗi bước
│   ├── diffs/               # Git diff THỰC TẾ sau khi áp dụng patch & merge
│   └── tests/               # Kết quả test (targeted, full-verification)
├── logs/                    # Workflow, command (redacted), llm prompts/responses
│   └── llm/                 # Lịch sử chi tiết các prompt/response LLM
└── pr/                      # PR summary markdown, urls, reviews
```

### 2. Các Cơ Chế Cốt Lõi

- **DB & Filesystem Reconciliation:** Khi resume task, hệ thống thực hiện đối chiếu (reconcile) giữa DB state (bảng `workflow_checkpoints` và `workflow_artifacts` của workflow engine) và thực hiện khôi phục (restore) lại các thay đổi/mã nguồn trong workspace filesystem (như apply lại các git patch của các bước đã hoàn thành trước đó) để đảm bảo đồng bộ trạng thái thực thi. Điều này giúp ngăn chặn triệt để tình trạng split-brain.
- **Distributed Locking:** File `.workspace.lock` hoạt động như một mirror lưu trữ (chứa owner ID, hostname, heartbeat, TTL) cho DB advisory/row lock, đảm bảo không bao giờ có hai worker cùng chạy một task và cho phép dọn dẹp stale lock.
- **Branch & Worktree Isolation:**
  - **Task Easy:** Giao tiếp trực tiếp với nhánh tích hợp (`feature/{task_id}`) bên trong thư mục `main/`, bỏ qua bước tạo role worktree để tối ưu tốc độ.
  - **Task Medium/Hard:** Hệ thống tự động cấp phát các Git worktrees riêng rẽ theo vai trò (VD: `worktrees/{role}/` cho frontend, backend) để các agent code song song. Sau đó merge tập trung lại nhánh integration tại `main/`.
- **Theo Dõi Trạng thái Từng Repo:** `metadata.json` ghi nhận riêng biệt `merge_status` và `test_status` cho từng repository. Tính năng này chặn đứng rủi ro bất đồng bộ tích hợp ở các cross-repo tasks (Ví dụ: repo A đã merge nhánh thành công nhưng repo B merge thất bại).
- **Dễ Dàng Debug qua Artifacts:** Các thay đổi chưa được commit khi dọn dẹp worktree sẽ được tự động lưu lại dưới dạng các file `cleanup-{repo}-{role}.diff` trong thư mục `artifacts/diffs/`, giúp quá trình audit và debug minh bạch mà không làm mất thông tin.
- **Dọn Dẹp An Toàn & Bảo Mật (Partial Cleanup):**
  - **Giữ lại dữ liệu Audit:** Khi task rơi vào trạng thái terminal (`merged` hoặc `failed`), hệ thống **tuyệt đối không** xóa toàn bộ thư mục root của task. Các thư mục `specs/`, `artifacts/` (checkpoints/diffs), `logs/`, `metadata.json` và `task.json` phải được giữ lại để phục vụ quá trình audit và debug.
  - **Kiểm tra Git State:** Chỉ tiến hành dọn dẹp các thư mục mã nguồn (`code/repos/*/worktrees/*`). Trước khi xóa một worktree, bắt buộc phải kiểm tra git state bằng `git status --porcelain`. Nếu có file chưa commit, phải lưu thành `artifacts/diffs/cleanup-{repo}-{role}.diff` trước khi tiến hành xóa worktree.
  - **Redaction:** Chức năng Redaction luôn chạy ngầm để bóc tách/loại bỏ token, secret, private keys khỏi các logs và AI prompts.
- **Path Standardisation & Context Sanitisation (Chuẩn hóa Đường dẫn & Bảo vệ Ngữ cảnh LLM):**
  - **PathManager làm Source of Truth:** Toàn bộ đường dẫn làm việc trên host và container được phân giải tập trung qua `PathManager` sử dụng cấu trúc chuẩn hóa `code/repos/`.
  - **Làm sạch Ngữ cảnh LLM (Option B):** Trước khi cung cấp danh sách file hoặc mã nguồn cho Agent (LLM), hệ thống sử dụng hàm `WorkspaceToRepoRelative` để che giấu tiền tố sandbox `code/repos/` và thông tin nhánh/worktree, nhưng vẫn giữ lại tên repository làm tiền tố (ví dụ: `repo-a/src/index.js`). Điều này vừa giúp bảo vệ cấu trúc nội bộ của sandbox, vừa ngăn chặn triệt để tình trạng trùng lặp/xung đột tên file giữa các repository trong dự án đa repo, đồng thời định hướng cho Agent viết đường dẫn chuẩn xác.
  - **Tự động khử tiền tố (Strip Prefix):** Khi Agent đề xuất thay đổi dạng patch/diff có kèm tiền tố đầy đủ, bộ lọc `CleanPatchPaths` và `SplitPatchByRepo` tự động tìm kiếm và loại bỏ phần tiền tố khớp với cấu hình của `PathManager` trước khi thực thi `git apply`, giúp tránh các lỗi lặp thư mục hoặc không nhận diện được file thay đổi.

### 3. Cấu Trúc Metadata & LLM Traces

**Task State (`task.json`):**
Snapshot của DB lưu trữ cấu hình cơ bản của task:
```json
{
  "task_id": "task-123",
  "project_id": "project-1",
  "title": "Implement user authentication",
  "description": "Add JWT based auth and login endpoints.",
  "status": "coding",
  "complexity": "medium",
  "spec_status": "approved",
  "labels": ["auth", "backend"]
}
```

**Routing & Integration State (`metadata.json`):**
Cấu hình định tuyến và tracking kết quả merge/test độc lập cho nhiều repo:
```json
{
  "workspace_version": 1,
  "repos": [
    {
      "id": "repo-1",
      "name": "backend-api",
      "status": {
        "merge_status": "merged", // pending, merged, conflict, failed, skipped
        "test_status": "pending"  // pending, passed, failed, skipped
      },
      "paths": {
        "main": "code/repos/backend-api/main",
        "worktrees": {
          "backend": "code/repos/backend-api/worktrees/backend",
          "fix": "code/repos/backend-api/worktrees/fix"
        }
      },
      "branches": {
        "integration": "feature/task-123",
        "backend": "feature/task-123-backend"
      }
    }
  ]
}
```

**LLM Trace Layout:**
Tất cả các lời gọi AI được ghi lại nguyên bản (đã redact secret) dưới thư mục `logs/llm/` theo thứ tự thời gian (giúp trace rõ các vòng lặp):
```text
logs/llm/call-{nnn}-{step}/
  ├── request.json
  ├── response.json
  ├── prompt.md
  ├── output.md
  ├── parsed.json
  └── metadata.json   # Lưu step, call_number, token usage, duration, retry count
```
