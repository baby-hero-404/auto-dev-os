## Workflow Chạy Như Thế Nào?

Khi một task được tạo, Workflow Engine sẽ tự động điều phối các Agent qua một chuỗi bước (DAG — Directed Acyclic Graph). Mỗi bước chỉ chạy khi các bước phụ thuộc đã hoàn thành:

```
Step 1: Phân tích (Analyze)
   Agent Planner phân tích task → tạo spec theo OpenSpec → đánh giá complexity.
   • Easy → tự động approve spec, nhảy thẳng sang coding.
   • Medium/Hard → DỪNG workflow → chờ con người review & approve spec.

Step 2: Lên kế hoạch (Plan)
   Chia task thành sub-tasks, xác định cần Agent nào (Backend, Frontend...).

Step 3: Viết code (Code)
   • Task đơn giản (Easy): 1 Agent Backend viết code.
   • Task phức tạp: Agent Backend + Agent Frontend viết code SONG SONG.

Step 4: Merge
   Gom kết quả từ các Agent viết code song song, kiểm tra conflict.

Step 5: Review chéo
   Agent Reviewer (khác Agent viết code) kiểm tra chất lượng → liệt kê findings.

Step 6: Sửa lỗi (Fix)
   Agent sửa các findings từ review. Nếu không có findings → skip bước này.

Step 7: Chạy test
   Tự động phát hiện và chạy test suite (Go test, npm test...) trong Docker sandbox.

Step 8: Tạo Pull Request
   Tạo branch → commit → push → tạo PR trên GitHub → lưu PR URL → DỪNG workflow.
   Con người review PR cuối cùng → Approve → Task hoàn thành.
```

## Workflow Thay Đổi Theo Độ Phức Tạp

| Complexity | Các bước thực thi | Đặc điểm |
|:-----------|:-----------------|:---------|
| **Easy** | Analyze → Code → Test → PR | Không plan, không review/fix, không frontend |
| **Medium** | Analyze → Plan → Code (BE+FE song song) → Merge → Review → Fix → Test → PR | Full pipeline, review-fix loop |
| **Hard** | Giống Medium + (Planned) Group Chat review | Deep review, cross-harness attestation |

## Cơ Chế Dừng & Tiếp Tục (Pause/Resume)

Workflow có 2 điểm dừng chờ con người:

| Khi nào dừng | Lý do | Tiếp tục khi |
|:-------------|:------|:-------------|
| Sau bước Analyze (Medium/Hard) | Chờ con người review spec/plan | Con người approve spec |
| Sau bước PR | Chờ con người review code | Con người approve PR |

Khi workflow dừng, toàn bộ trạng thái được lưu (checkpoint). Khi tiếp tục, workflow chạy tiếp từ đúng bước kế tiếp — không cần chạy lại từ đầu.

## Checkpoint & Recovery

*   **Checkpoint:** Sau mỗi bước thành công, kết quả được lưu lại. Nếu workflow bị gián đoạn (server restart, lỗi tạm thời), hệ thống tự động tải lại các checkpoint đã lưu và **resume từ bước thành công gần nhất** mà không phải chạy lại từ đầu.
*   **Artifacts:** Code patches, diffs, review findings, test results đều được lưu trữ — giúp con người kiểm tra lại bất kỳ lúc nào.

## Error Handling

**Các cơ chế đã hiện thực (Implemented):**
*   **Auto-retry:** Tự động chạy lại toàn bộ DAG khi gặp lỗi thực thi (không phải lỗi tạm dừng). Số lần chạy tối đa được xác định bởi cấu hình `max_retries` của Project.
*   **Recovery / Restart:** Task ở trạng thái `failed` có thể gọi API `POST /api/v1/tasks/{taskID}/restart` để kích hoạt lại. Hệ thống sẽ tự động tải các checkpoint trước đó và resume từ bước lỗi cuối cùng.
*   **Review-fix limit:** Giới hạn số vòng lặp review → fix để tránh lặp vô tận. Hệ thống đọc cấu hình `max_review_fix_cycles` của Project (mặc định là 3). Khi vượt quá giới hạn này, bước `review` sẽ tự động bỏ qua việc sửa lỗi và đẩy tiếp sang bước `testing`.

**Planned hardening:**
*   **Notification:** Task `failed` → thông báo cho người tạo task qua Web UI (và chat nếu cấu hình §5.10).
