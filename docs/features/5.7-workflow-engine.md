# 5.7. Workflow Engine

**Status:** Implemented / Planned multi-agent expansion  
**Owner docs:** `docs/ARCHITECTURE.md`; `docs/features/5.3-agent-system.md` for orchestration patterns  
**Code areas:** `server/internal/workflow`, `server/internal/orchestrator`, `server/internal/sandbox`, `server/internal/gitops`  
**Blocking decisions:** Durable workflow engine choice for future scale: keep current engine, Temporal, or LangGraph-style graph orchestration.  
**Acceptance criteria:** Task can progress through context loading, analysis, planning, coding, review, fix, testing, PR creation, and human review with persisted workflow state and bounded review loops.

**Mục tiêu:** Tự động hóa toàn bộ quy trình phát triển phần mềm — từ phân tích yêu cầu đến tạo Pull Request — bằng một workflow engine có khả năng chạy song song, tạm dừng chờ con người, và tự phục hồi khi gặp lỗi.

---

## Workflow Chạy Như Thế Nào?

Khi một task được tạo, Workflow Engine sẽ tự động điều phối các Agent qua một chuỗi bước (DAG — Directed Acyclic Graph). Mỗi bước chỉ chạy khi các bước phụ thuộc đã hoàn thành:

```
Step 0: Nạp ngữ cảnh (Context Load)
   Agent checkout repository, đọc:
   • Cấu trúc repo, branch hiện tại, recent commits
   • Issue/task description, linked issues
   • Coding conventions (linter config, .editorconfig, CONTRIBUTING.md)
   • Test commands (Makefile, package.json scripts)
   • CI config (GitHub Actions, GitLab CI)
   • ARCHITECTURE.md, relevant domain docs
   → Mục tiêu: Agent có đủ ngữ cảnh trước khi phân tích, tránh hallucination.

Step 1: Phân tích + Spec + Complexity (Analyze)
   Agent Planner phân tích task → tạo spec theo OpenSpec → đánh giá complexity.
   Spec bao gồm: scope, affected files, risks, test strategy, execution plan.
   → Output: spec document + complexity label (Easy/Medium/Hard) + risk assessment.

Step 2: Human Gate (có điều kiện)
   • Easy + Low-risk → auto-approve spec, tiếp tục workflow.
   • Easy + High-risk → DỪNG workflow → chờ con người review.
   • Medium/Hard → LUÔN dừng → chờ con người review & approve.
   Con người thấy: spec, scope, plan, affected files, risks, test strategy.
   (Xem bảng "High-risk domains" bên dưới.)

Step 3: Lên kế hoạch thực thi (Plan Execution)
   Chia task thành sub-tasks với ownership rõ ràng.
   Xác định Agent nào phụ trách phần nào (Backend, Frontend...).
   Nếu song song: mỗi Agent có worktree/branch riêng, tránh conflict.

Step 4: Viết code (Code)
   • Task đơn giản: 1 Agent viết code trên feature branch.
   • Task song song: Agent Backend + Agent Frontend viết code trên branch riêng.
     Backend: feature/{task-id}-be
     Frontend: feature/{task-id}-fe
   → Sau khi code xong, chạy targeted tests liên quan (unit tests của files đã sửa).

Step 5: Merge / Integrate
   • Task đơn giản: không cần merge (đã trên 1 branch).
   • Task song song: merge các branch riêng vào integration branch feature/{task-id}.
     Kiểm tra conflict. Nếu conflict → Agent tự resolve hoặc escalate.

Step 6: Agent Review (Cross-review)
   Agent Reviewer (khác Agent viết code) kiểm tra chất lượng → liệt kê findings.
   Review checklist: code quality, adherence to spec, test coverage, security.

Step 7: Sửa lỗi (Fix)
   Agent sửa các findings từ review. Nếu không có findings → skip bước này.
   → Sau fix, chạy lại targeted tests trên các files đã sửa.

Step 8: Re-review (có điều kiện)
   Nếu fix ở Step 7 thay đổi đáng kể (>N lines hoặc >M files) → quay lại Step 6.
   Vòng lặp Review → Fix → Re-review giới hạn tối đa max_review_fix_cycles
   (mặc định: 3 vòng). Khi vượt quá → chuyển sang Step 9 + đánh dấu cảnh báo.

Step 9: Chạy tests + lint + build
   Chạy full relevant test suite trong Docker sandbox:
   • Unit tests + integration tests liên quan
   • Linting (golangci-lint, eslint...)
   • Build verification (go build, npm run build)
   → Nếu fail → quay lại Step 7 (nằm trong giới hạn review-fix cycles).

Step 10: Tạo Pull Request
   Tạo branch → commit → push → tạo PR trên GitHub → lưu PR URL.
   PR description bao gồm: summary, affected files, test results, risk notes.
   → Workflow chuyển sang trạng thái PR_READY — task CHƯA hoàn thành.

Step 11: Human Final Review
   Con người review PR cuối cùng trên GitHub/Web UI.
   • Approve → chuyển sang Step 12.
   • Request changes → workflow quay lại Step 7 (trong giới hạn).
   • Reject → task chuyển status failed, lưu lý do.

Step 12: Merge & Complete
   Merge PR vào target branch.
   Task status → merged. Task hoàn thành.
```

## High-Risk Domains (Auto-approve Blocked)

Dù task được đánh giá là **Easy**, nếu task chạm vào bất kỳ domain nào dưới đây thì spec **không được auto-approve** — phải qua human gate:

| Domain | Ví dụ |
|:-------|:------|
| **Authentication / Authorization** | Login flow, JWT, OAuth, session management |
| **Payment / Billing** | Stripe integration, invoice logic, pricing |
| **Data Migration** | Schema changes, data backfill, ETL scripts |
| **Infrastructure** | Dockerfile, docker-compose, CI/CD config, deployment |
| **Security** | Encryption, secrets, CORS, CSP, vulnerability fixes |
| **Permission / RBAC** | Role changes, access control, policy engine |
| **Public API Contract** | Breaking changes to REST/gRPC endpoints, schema versioning |

Agent Planner phải tag `risk_domains` trong spec output. Orchestrator kiểm tra danh sách này trước khi quyết định auto-approve hay dừng tại human gate.

## Workflow Thay Đổi Theo Độ Phức Tạp

| Complexity | Các bước thực thi | Đặc điểm |
|:-----------|:-----------------|:---------|
| **Easy (low-risk)** | Context → Analyze → Code → Targeted Test → Full Test → PR → Human Review → Merge | Auto-approve spec, single agent, no plan step |
| **Easy (high-risk)** | Context → Analyze → **Human Gate** → Code → Targeted Test → Full Test → PR → Human Review → Merge | Spec needs human approval despite easy complexity |
| **Medium** | Context → Analyze → **Human Gate** → Plan → Code (có thể song song) → Merge → Review → Fix → Re-review → Full Test → PR → Human Review → Merge | Full pipeline, bounded review-fix loop |
| **Hard** | Giống Medium + Cross-harness attestation + (Planned) Group Chat review | Deep review, multi-agent cross-validation |

## Parallel Coding & Ownership

Khi task cần cả Backend và Frontend song song:

```
                    Plan (Step 3)
                        │
              ┌─────────┴─────────┐
              ▼                   ▼
     Agent Backend           Agent Frontend
     branch: feature/        branch: feature/
       {task-id}-be            {task-id}-fe
     worktree riêng          worktree riêng
              │                   │
              └─────────┬─────────┘
                        ▼
              Integration Merge (Step 5)
              branch: feature/{task-id}
                        │
                        ▼
                 Review → Fix → Test → PR
```

**Nguyên tắc ownership:**
- Mỗi Agent chỉ sửa files trong phạm vi ownership đã xác định ở Step 3.
- Shared files (e.g., API contracts, types) → chỉ 1 Agent sửa, Agent còn lại đọc.
- Conflict khi merge → Agent tự resolve dựa trên ownership. Nếu không resolve được → escalate lên human gate.

## Review-Fix Cycle Limits

Vòng lặp Review → Fix → Re-review **có giới hạn** để tránh workflow loop vô hạn:

```
max_review_fix_cycles = 3  (project-level config)

Review ──► Findings? ──► Yes ──► Fix ──► Re-review
  ▲              │                          │
  │              No                    cycle_count++
  │              │                          │
  │              ▼                   cycle > max?
  │           Skip Fix,                     │
  │           go to Test              Yes ──► STOP loop,
  │                                          go to Test
  │                                          + WARNING flag
  └──────── No (under limit) ◄──────────────┘
```

Khi vượt giới hạn, task được đánh dấu `review_limit_exceeded: true` trong metadata. Human reviewer sẽ thấy cảnh báo này trong PR.

## Testing Strategy

Tests được chạy **nhiều lần** tại các thời điểm khác nhau:

| Thời điểm | Loại test | Mục đích |
|:-----------|:----------|:---------|
| Sau Step 4 (Code) | Targeted tests — unit tests của files đã sửa | Phát hiện lỗi sớm trước khi review |
| Sau Step 7 (Fix) | Targeted tests — unit tests của files đã fix | Đảm bảo fix không gây regression |
| Step 9 (Pre-PR) | Full relevant test suite + lint + build | Gate cuối trước khi tạo PR |

## Cơ Chế Dừng & Tiếp Tục (Pause/Resume)

Workflow có tối đa 3 điểm dừng chờ con người:

| Khi nào dừng | Lý do | Tiếp tục khi |
|:-------------|:------|:-------------|
| Sau Step 1 (Medium/Hard hoặc Easy + high-risk) | Chờ con người review spec, plan, risks | Con người approve spec |
| Sau Step 10 (PR created) | Chờ con người review code trên PR | Con người approve PR |
| Sau conflict escalation (Step 5, nếu có) | Merge conflict không tự resolve được | Con người resolve conflict |

Khi workflow dừng, toàn bộ trạng thái được lưu (checkpoint). Khi tiếp tục, workflow chạy tiếp từ đúng bước kế tiếp — không cần chạy lại từ đầu.

## Task Completion Policy

> **PR ≠ Task hoàn thành.**

| Trạng thái | Ý nghĩa |
|:-----------|:---------|
| `pr_ready` | PR đã tạo, chờ human review. Task **chưa** hoàn thành. |
| `human_review` | Con người đang review PR. |
| `merged` | PR đã được merge. Task **hoàn thành**. |
| `failed` | Task thất bại (reject, quá giới hạn retry, lỗi không recover). |

Task chỉ chuyển sang `merged` (hoàn thành) khi: con người explicitly approve PR **và** merge thành công vào target branch.

## Checkpoint & Recovery

*   **Checkpoint:** Sau mỗi bước thành công, kết quả được lưu lại. Nếu workflow bị gián đoạn (server restart, lỗi tạm thời), hệ thống tự động tải lại các checkpoint đã lưu và **resume từ bước thành công gần nhất** mà không phải chạy lại từ đầu.
*   **Artifacts:** Code patches, diffs, review findings, test results, spec documents đều được lưu trữ — giúp con người kiểm tra lại bất kỳ lúc nào.

## Error Handling

**Các cơ chế đã hiện thực (Implemented):**
*   **Auto-retry:** Tự động chạy lại toàn bộ DAG khi gặp lỗi thực thi (không phải lỗi tạm dừng). Số lần chạy tối đa được xác định bởi cấu hình `max_retries` của Project.
*   **Recovery / Restart:** Task ở trạng thái `failed` có thể gọi API `POST /api/v1/tasks/{taskID}/restart` để kích hoạt lại. Hệ thống sẽ tự động tải các checkpoint trước đó và resume từ bước lỗi cuối cùng.
*   **Review-fix limit:** Giới hạn số vòng lặp review → fix, cấu hình qua `max_review_fix_cycles` (mặc định: 3). Khi vượt giới hạn → skip fix, đẩy sang testing + đánh dấu cảnh báo.

**Planned hardening:**
*   **Notification:** Task `failed` → thông báo cho người tạo task qua Web UI (và chat nếu cấu hình §5.10).
*   **Timeout per step:** Giới hạn thời gian tối đa cho mỗi bước workflow, tránh treo vô hạn.
