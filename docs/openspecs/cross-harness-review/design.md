# Design: Cross-Harness Review

## Resolver

```go
// engine-aware harness resolver, đặt cạnh model-level resolver hiện có
func ResolveReviewHarness(policy string, codedBy HarnessID, available []ProviderModels) (HarnessID, []string /*warnings*/)
```

Thuật toán: different_provider → provider ≠ codedBy.Provider đầu tiên có model đủ cấp (map model-level của project); different_model → cùng provider, model khác; degrade chain như REQ-001.

**CLI-coded task — blind-spot guard**: `cli:<command>` KHÔNG tự động "khác mọi thứ". `cli_engine_config` có field optional `underlying_provider` (anthropic/openai/google/...) — user khai CLI của họ chạy model nhà nào (Claude Code → anthropic). Resolver dùng `effectiveProvider = underlying_provider ?? "cli:"+command`:
- Có underlying → chọn review provider **khác underlying** (Claude Code + reviewer Claude cùng chia sẻ blind spots của cùng base model — mục đích cross-review mất tác dụng).
- Không khai → hành vi cũ + log khuyến nghị.

**Adversarial directive**: khi degrade kết thúc ở same provider (resolver trả flag `sameHarness=true`), review prompt được thêm section:

```
## Adversarial audit mode
You are auditing code generated by another AI system. Do not assume its logic is
correct or that its tests are sufficient. Actively hunt for: off-by-one and boundary
errors, swallowed errors / ignored returns, nil/undefined dereference, injection and
unsafe input handling, race conditions, and spec requirements silently skipped.
```

Chỉ inject khi sameHarness — provider thực sự khác thì directive là nhiễu.

## Metadata

Step state keys `coded_by`, `reviewed_by` (JSON `{engine, provider, model}`); PR step đọc từ state đưa vào description footer:

```
---
Coded by: cli:claude (subscription) | Reviewed by: api:openai/gpt-x
```

Đây là nền attestation (P4.3) — set đó sẽ ký/đóng gói DSSE, set này chỉ ghi metadata.

## cross_review step (CLI flow)

- Input: `git diff` của worktree (qua tool có sẵn) + spec set files + 2-verdict prompt của `review-verdict-split` (tái dùng `review.md` với input assembly khác — extract phần chung ra template partial nếu cần).
- Routing fail: re-dispatch cli_implement với violations trong prompt (`## Reviewer feedback` — cùng mẫu với spec-approval request_changes của Wave 1); dùng chung đếm `MaxReviewFixCycles`.
- DAG: node optional — build definition thời điểm resolve engine, policy≠same thì chèn node (BuildWorkflow đã nhận engine từ Wave 1; thêm project vào signature nếu chưa).

## Trade-offs

- Áp mềm (degrade + warning) thay vì enforce cứng như ai-sdlc DSSE: không chặn user chỉ có 1 API key; enforce cứng để attestation P4.3 cân nhắc.
- Repeat-violation escalation (review-verdict-split REQ-003) áp dụng luôn cho cross_review — cùng cơ chế state.
