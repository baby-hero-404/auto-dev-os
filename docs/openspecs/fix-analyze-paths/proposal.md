# Proposal: Fix Analyze Tool Path Resolution

## Why
The LLM Planner context generated by the `AnalyzeStep` displayed the root path as `.` and included a noisy list of non-repository workspace files (e.g., `.workspace.lock`, `metadata.json`, `logs/`). This was caused by two issues:
1. Manual string manipulation instead of using the `PathManager` value objects, leading to improper resolution contexts when multiple tools are run.
2. A mock `s.sandbox.RunCommand(..., "find .")` executed inside the global workspace root, confusing the LLM by polluting logs with unrelated files, while the actual `list_files` tool returned `.` because the newly checked-out repo was completely empty except for a filtered `.git` directory.

## What Changes

### Issue 1: Manual Path Resolution
- Change `listAnalyzeFiles` to use `pkg/paths.OSWorkspacePaths` to resolve task and repository roots.
- Change `readAnalyzeFile` to use `pkg/paths.OSWorkspacePaths`.
- Change `grepAnalyzeFiles` to use `pkg/paths.OSWorkspacePaths`.
- Change `executeAnalyzeTool` to use `pkg/paths.OSWorkspacePaths` instead of manual `filepath.Join()`.

### Issue 2: Sandbox Mock Logging Noise
- Remove the `s.sandbox.RunCommand` mock `find .` from `listAnalyzeFiles`.
- Remove the `s.sandbox.RunCommand` mock `grep -RIn` from `grepAnalyzeFiles`.

## Capabilities

### Modified Capabilities
- `AnalyzeStep`: Correctly limits scope strictly to `code/repos/<repo>/main` using `OSWorkspacePaths` and avoids unnecessary mock command executions that pollute tool output.
