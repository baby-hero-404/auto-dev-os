# Architecture Decision Record: LLM Patching Robustness

**Date:** June 29, 2026  
**Status:** Accepted  

## 1. Problem Statement

During the `code_backend` workflow step, an invalid patch generated by the coding agent caused `git apply` to fail with exit code 2. Because the orchestrator treated patch application as a fatal error, the workflow terminated immediately. This abrupt crash prevented the system from reaching the Test and Review stages, disabling the orchestrator's built-in self-healing loop.

## 2. Technical Analysis

The issue stems from a combination of non-deterministic generation, deterministic verification, and a brittle workflow design.

### 2.1. Invalid Patch Generation
The immediate trigger was an invalid patch generated by the coding agent. The patch contained multiple structural violations:
- **Missing repository prefix:** The diff targeted `a/readme.md` instead of the expected Option B format `a/test/readme.md`.
- **Incorrect hunk metadata:** The hunk header (`@@ -1,83 +1,118 @@`) declared 83 original lines, but the target file contained 84 lines.
- **Patch syntax violation:** The mismatch between the header and the actual content length rendered the unified diff corrupt.

### 2.2. Deterministic Verification (`git apply`)
The `git apply` utility correctly rejected the patch. It operates mathematically and deterministically; because the hunk metadata no longer matched the original file, it returned `exit code 2` (unexpected end of file/corrupt patch) to prevent unintended code corruption. 

### 2.3. Workflow Design Flaw
The primary systemic root cause was the orchestrator's error-handling strategy:
- **Flawed Flow:** `Patch Failure` ➔ `Fatal Error` ➔ `Workflow Exit`
- **Ideal Flow:** `Patch Failure` ➔ `Recoverable Event` ➔ `Review` ➔ `Retry`

By failing immediately, the system denied the LLM the opportunity to learn from the patch failure and regenerate a correct response.

## 3. Immediate Fixes

To stabilize the workflow, we deployed the following tactical changes to the `server/internal/orchestrator` package:

| Change | Purpose |
| :--- | :--- |
| **Graceful Degradation** | Prevent workflow termination by logging patch failures as `warn` and continuing to the Review step. |
| **Empty Repo Fallback** | Improve robustness by inferring the repository name if missing in a single-repo workspace, preventing malformed step IDs. |
| **Prompt Constraints** | Reduce the frequency of malformed patches by explicitly defining unified diff requirements in the system instruction. |

## 4. Lessons Learned

- **Patch generation should not be considered deterministic.** LLMs are probabilistic text generators, not mathematical diff engines.
- **LLM outputs require validation before execution.** Trusting the LLM to produce flawlessly formatted system commands leads to brittle architecture.
- **Apply failures should be treated as recoverable events whenever possible.** The orchestrator must prioritize continuity over strictness.
- **Reviewers should participate even after patch failures.** A failed patch means no changes were made; the reviewer is perfectly positioned to instruct the coding agent to try again.

## 5. Future Architecture

Relying exclusively on Unified Diffs (`git apply`) restricts the orchestrator's reliability. We recommend evaluating alternative editing strategies as our preferred direction for future development.

| Strategy | Advantages | Limitations |
| :--- | :--- | :--- |
| **Unified Diff** | Standardized, natively compatible with Git | Highly sensitive to line numbers and whitespace drift |
| **Search & Replace** | Robust against line drift, natural for LLMs | Requires unique search blocks to prevent misapplication |
| **AST Editing** | Semantically aware, precise scoped changes | Language-dependent, requires heavy parsing infrastructure |

## 6. Hybrid Architecture Proposal

Instead of hardcoding a single patching method, the orchestrator should implement a **Patch Engine Abstraction**. This allows the system to validate patches and choose the most appropriate application strategy dynamically.

```text
LLM Generation
      ↓
Patch Validation (Syntax, Metadata, Uniqueness)
      ↓
Choose Patch Strategy
  ├── Search & Replace (Fallback/Fuzzy)
  ├── Unified Diff (Primary if valid)
  └── AST Edit (Semantic scopes)
      ↓
    Apply
      ↓
     Test
      ↓
   Reviewer
      ↓
    Retry (If validation or apply fails)
```

By isolating the `Patch Strategy` into an abstraction layer, the orchestrator's workflow remains decoupled from the specific editing engine.

## 7. Architectural Decisions

**Short-term (Implemented)**
- [x] Patch failures become recoverable events.
- [x] Continue workflow to Reviewer on patch failure.
- [x] Improve LLM prompts to enforce stricter diff formatting.

**Medium-term**
- [ ] Introduce a dedicated **Patch Validator** before execution (Verify repo exists, syntax, metadata, no duplicated hunks).
- [ ] Introduce the **Search & Replace** block format as an alternative editing strategy.

**Long-term**
- [ ] Support multiple edit strategies through a common `Patch Engine` interface, allowing the orchestrator to dynamically select the safest editing method.
